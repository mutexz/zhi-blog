# 知行小站：部署与写作指南

本文说明如何把站点部署到线上，以及新增**长文**、**笔记**时应使用的文件位置与 frontmatter 模板。

---

## 一、部署前必读


| 变量              | 何时必填             | 含义                                                                                                             |
| --------------- | ---------------- | -------------------------------------------------------------------------------------------------------------- |
| `**SITE`**      | 每次**生产构建**       | 站点绝对地址，无末尾斜杠。例：`https://my-blog.vercel.app` 或 `https://blog.example.com`。用于 RSS、sitemap、Open Graph 等绝对链接。      |
| `**BASE_PATH`** | 仅部分 GitHub Pages | 仓库为**项目站点**（地址形如 `https://user.github.io/repo-name/`）时，设为 `/<repo-name>/`（前后都要有 `/`）。根站点 `user.github.io` 不要设。 |


本地开发可不设 `SITE`，默认使用 `astro.config.mjs` 里的占位域名；**上线前必须在托管平台配置 `SITE` 并重新部署**。

## 二、推荐方案：Vercel（与官方一致，2026）

官方说明：

- Vercel：[Astro on Vercel](https://vercel.com/docs/frameworks/astro)
- Astro：[Deploy your Astro Site to Vercel](https://docs.astro.build/en/guides/deploy/vercel/)

**静态 Astro 站点**在 Vercel 上属于 **零额外框架配置**：连接 Git 后一般会 **自动识别 Astro**，无需手写 `vercel.json` 即可部署。本仓库为 **纯静态输出**（`output: 'static'`），**不必**安装 `@astrojs/vercel`，除非你以后要 **SSR/按需渲染**、或要用官方文档里写的 **Vercel Web Analytics / Image Optimization** 等（见 Vercel 文档 [Astro on Vercel](https://vercel.com/docs/frameworks/astro) 中 *Using Vercel's features with Astro* 一节）。

### 2.1 前提

- 代码已在 **GitHub / GitLab / Bitbucket**（Vercel 支持这些 Git 集成，见 [Git Integration](https://vercel.com/docs/concepts/git)）。
- 确认你的仓库是下面两种布局中的哪一种，**Root Directory 只能二选一**。

### 2.2 仓库布局与 Root Directory

| 布局 | 说明 | **Root Directory**（Vercel 项目设置） |
| --- | --- | --- |
| **A. 独立博客仓库** | 仓库根目录就是 Astro 项目（例如 GitHub `mutexz/zhi-blog`，根目录含 `package.json`、`astro.config.mjs`） | **留空**（或 `.`，表示仓库根） |
| **B. 大仓库里的子目录** | 博客在 `zhi-blog/` 下，与别的项目同级 | 填 **`zhi-blog`** |

导入后若构建失败，到 **Settings → General → Root Directory** 核对是否与上表一致。

### 2.3 导入项目

1. 打开 [vercel.com/new](https://vercel.com/new)（或控制台 **Add New… → Project**）。
2. 登录并 **Import** 你的 Git 仓库。
3. 按 **2.2** 设置 **Root Directory**，其余多数可保持默认。

### 2.4 构建设置（与官方默认一致）

Vercel 对 Astro 的默认行为与 Astro 文档一致：**Build** 一般为 `npm run build`（即 `astro build`），输出目录为 **`dist`**。

| 项 | 建议值 |
| --- | --- |
| **Framework Preset** | **Astro**（通常会自动检测） |
| **Build Command** | `npm run build` |
| **Output Directory** | `dist` |
| **Install Command** | 默认 `npm install` 即可 |
| **Node.js 版本** | 本仓库 `package.json` 中 `engines.node` 为 `>=20.3.0`；可在 Vercel **Settings → General → Node.js Version** 指定 **20.x**，与官方对现代 Astro 的要求一致 |

若向导询问是否覆盖默认设置，**优先保留自动检测**（与官方「Want to override the settings? 选 N」一致）。

### 2.5 首次部署

点击 **Deploy**。完成后会得到 `https://<项目名>.vercel.app` 一类域名。**生产环境**下请继续 **2.6**，把 `SITE` 配成该域名（或你的自定义域名），否则 RSS / sitemap / 绝对 URL 仍可能是占位值。

### 2.6 环境变量 `SITE` 并重新部署

本站 `astro.config.mjs` 使用 **`process.env.SITE`** 作为 `site`（见第一节表格）。Vercel 上需在控制台注入该变量，构建时才会写入正确绝对地址：

1. 项目 **Settings → Environment Variables**。
2. 新增：**Name** `SITE`，**Value** `https://你的.vercel.app` 或自定义域（**无**末尾 `/`）。
3. 作用域至少勾选 **Production**（建议 **Preview** 一并勾选，便于预览链接正确）。
4. **Deployments** → 最新一条 **⋯ → Redeploy**（若需确保重新跑完整构建，可关闭 **Use existing Build Cache**）。

### 2.7 绑定自定义域名（可选）

1. **Settings → Domains** → 添加域名，按提示配置 DNS（**CNAME** / **A**）。
2. 将 **`SITE`** 更新为 `https://你的域名`，再 **Redeploy**。

自定义域往往比默认 `*.vercel.app` 在部分网络环境下 **更易访问**（见 **2.8**）。

### 2.8 浏览器无法打开 `*.vercel.app`？

部署成功（Vercel 控制台为 **Ready**）但本机提示「无法连接服务器」，多为 **本地网络 / DNS / 公司策略** 无法访问 Vercel 默认子域，**不一定是项目配错**。可尝试：

- 换 **手机蜂窝网络** 或其它网络验证；
- 更换 DNS（如 `1.1.1.1`、`8.8.8.8`）；
- 使用 **2.7 自定义域名**；
- 终端执行 `ping`、`nslookup zhi-blog.vercel.app` 辅助判断解析与连通性。

### 2.9 CLI 部署（可选）

安装 [Vercel CLI](https://vercel.com/cli) 后，在项目根执行 `vercel` / `vercel --prod`。官方说明见 Astro 文档 [CLI deployment](https://docs.astro.build/en/guides/deploy/vercel/#cli-deployment)。仍需在 Vercel 项目里配置 **`SITE`** 等环境变量。

### 2.10 进阶：`vercel.json`

若需响应头、重定向等，可使用官方 [Project Configuration](https://vercel.com/docs/project-configuration)（`vercel.json`）。本博客默认 **不需要**。

## 三、备选：GitHub Pages

### 3.1 启用 Pages

1. 仓库 **Settings → Pages**。
2. **Build and deployment → Source** 选择 **GitHub Actions**（不要选 Deploy from a branch 来传 `dist`，除非你自己改流程）。

### 3.2 配置仓库变量

**Settings → Secrets and variables → Actions → Variables**（Repository variables）：


| 变量              | 用户站点示例                    | 项目站点示例                          |
| --------------- | ------------------------- | ------------------------------- |
| `**SITE`**      | `https://<用户名>.github.io` | `https://<用户名>.github.io/<仓库名>` |
| `**BASE_PATH`** | 不创建或留空（默认 `/`）            | `/<仓库名>/`                       |


说明：用户站点指仓库名为 `<用户名>.github.io` 且内容为站点根；项目站点指普通仓库，访问路径带仓库名。

### 3.3 触发部署

**仅适用于「大仓库 + 博客在 `zhi-blog/` 子目录」**：向 `**main`** 分支推送且变更路径包含 `**zhi-blog/****` 时，工作流 `**.github/workflows/deploy-zhi-blog.yml**` 才会按路径过滤触发（以仓库内实际文件名为准）。若博客为 **独立仓库**（根目录即 Astro），需把工作流与检出目录改成以仓库根为项目根，或改用 **第二节 Vercel**。

### 3.4 首次使用 Actions 部署 Pages

若提示没有 **github-pages** 环境，按 GitHub 引导授权一次即可。

## 四、本地开发与生产构建

```bash
cd zhi-blog
npm install
npm run dev
```

浏览器访问终端中提示的地址（多为 `http://127.0.0.1:4321`）。

生产构建：

```bash
cd zhi-blog
# Linux / macOS
export SITE=https://你的线上域名
npm run build
```

或使用 `.env`（勿把含密钥的 `.env` 提交到 Git；`SITE` 可提交 `.env.example` 作说明）：

```bash
cp .env.example .env
# 编辑 .env 写入 SITE=...
npm run build
```

产物在 `**dist/**`，可上传到任意静态托管。

## 五、新增长文（文章）

### 5.1 存放位置

在目录 `**src/content/articles/**` 下新建 `**.md` 或 `.mdx**` 文件。

**URL 规则**：文件名（不含扩展名）即为路径段。  
例：`my-first-post.md` → `https://你的域名/articles/my-first-post/`

建议使用**英文小写、连字符**文件名，避免 URL 编码问题：如 `ai-eval-guide.md`。

### 5.2 Frontmatter 模板（复制后改内容）

```yaml
---
title: "文章标题（可含标点）"
description: "列表页与 SEO 用的短摘要，一两句话。"
pubDate: 2026-04-05
# 可选：更新日期
# updatedDate: 2026-04-06
tags:
  - 标签甲
  - 标签乙
readTime: 约 15 分钟
featured: false
# 可选：首页「精选长文」会优先展示 featured: true（仍按日期排序取前若干篇）
# featured: true
# 可选：封面图（需将图片放在 src/assets/ 并在 frontmatter 用相对路径引用，进阶用法见 Astro 文档）
# heroImage: ../../assets/cover.jpg
---

正文从这里开始，使用标准 Markdown。

## 二级标题

- 列表项
```

### 5.3 字段说明


| 字段            | 必填  | 说明                                  |
| ------------- | --- | ----------------------------------- |
| `title`       | 是   | 文章标题。                               |
| `description` | 是   | 摘要。                                 |
| `pubDate`     | 是   | 发布日期，`YYYY-MM-DD`。                  |
| `updatedDate` | 否   | 更新日期。                               |
| `tags`        | 否   | 字符串数组；文章列表页可按标签筛选。                  |
| `readTime`    | 否   | 展示用阅读时长文案。                          |
| `featured`    | 否   | `true` 时有机会出现在首页精选（与日期等逻辑共同作用）。     |
| `heroImage`   | 否   | 封面图（需符合 Astro Content 对 image 的配置）。 |


### 5.4 正文建议

- 正文**不要**再写一级 `# 标题`，页面已有标题，避免重复。
- 表格、代码块、引用等使用 GFM 语法即可；表格在站点内已做基础样式与横向滚动。
- **子路径部署（`BASE_PATH`）时**：若在 Markdown 里手写站内链接，请使用**相对路径**（如 `[另一篇](./other-post/)`）或**完整站外 URL**，避免写死以 `/` 开头的根路径（如 `[错](/articles/foo/)`），否则可能缺少仓库子路径前缀。导航栏与列表由模板自动生成，无需担心。

## 六、新增笔记（短文）

### 6.1 存放位置

在 `**src/content/notes/`** 下新建 `**.md` 或 `.mdx`**。

**URL**：`文件名.md` → `/notes/文件名/`

### 6.2 Frontmatter 模板

```yaml
---
title: "笔记标题"
description: "一行摘要，会显示在列表里。"
pubDate: 2026-04-05
noteType: 摘录
---

正文，可长可短。
```

### 6.3 `noteType` 取值（必填，且只能是以下之一）


| 值     | 用途建议       |
| ----- | ---------- |
| `摘录`  | 读书/文章摘录    |
| `链接`  | 以链接与短评为主   |
| `短想法` | 几句话的观点     |
| `工具`  | 工具、命令、配置备忘 |


（值须与 `src/content.config.ts` 中枚举一致，不要随意新造，否则构建会报错。若需新类型，要改配置里的 `z.enum`。）

### 6.4 RSS 说明

当前 RSS **仅包含长文**（`articles`）。笔记不会进入 `rss.xml`；若需要可把需求记作后续改进。

## 七、发布后检查清单

- 打开首页、文章列表、单篇文章、笔记列表、单篇笔记。
- 打开 `/rss.xml`，确认链接为 `https://你的域名/...`。
- 浏览器开发者工具或 [分享调试工具](https://developers.facebook.com/tools/debug/) 抽查 Open Graph（可选）。
- 手机端看表格是否可横向滑动、标题 hover 是否正常。

## 八、常见问题

**Q：改完 Markdown 本地看不到？**  
A：保存文件后 dev 一般会自动刷新；若未刷新，重启 `npm run dev`。

**Q：构建报错 frontmatter？**  
A：对照第五节、第六节检查缩进（YAML 对缩进敏感）、引号、必填字段。

**Q：GitHub Pages 样式全挂、资源 404？**  
A：多半是项目站未设置 `**BASE_PATH=/<仓库名>/`**，或 `SITE` 与真实访问 URL 不一致。
